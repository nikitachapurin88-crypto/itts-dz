# Sorting Visualizer — финальный отчёт по проекту

Учебный проект курса "Введение в специальность". Веб-приложение для визуализации алгоритмов
сортировки, разработанное с прохождением полного цикла: от идеи до
контейнеризации, тестирования, статического анализа и анализа зависимостей.

---

## 1. Описание системы

**Что это.** Веб-приложение, которое наглядно показывает работу алгоритмов
сортировки в реальном времени. Пользователь выбирает алгоритм, задаёт размер
и тип входного массива, запускает анимацию и наблюдает каждый шаг: сравнения,
перестановки (свапы), элементы, вставшие на финальные позиции.

**Зачем.** Алгоритмы сортировки обычно изучают по псевдокоду — трудно
прочувствовать, чем bubble отличается от quick по числу операций. Визуализация
делает разницу очевидной: видно, как растут счётчики сравнений и свапов на
одних и тех же данных.

**Функционал.**
- 7 алгоритмов: Bubble, Insertion, Selection, Quick, Merge, Shell, Heap.
- 3 типа входного массива: случайный, обратно отсортированный, почти
  отсортированный.
- Настройка размера массива (5–100 элементов) и скорости анимации.
- Счётчики сравнений и свапов в реальном времени.
- Цветовая индикация: сравниваемые элементы, свапы, финальные позиции.

## 2. Архитектура

### Принцип
Логика и отображение полностью разделены. Бэкенд не знает про анимацию —
он принимает массив и название алгоритма и возвращает **список всех шагов**
сортировки. Фронтенд проигрывает эти шаги с заданной скоростью. Это значит,
что алгоритмы можно тестировать без UI, а UI менять без правок алгоритмов.

### Структура
```
itts-dz/
├── app.py              # Flask-приложение, HTTP-роуты
├── algorithms.py       # Алгоритмы сортировки (генерируют шаги)
├── requirements.txt    # Зависимости
├── Dockerfile          # Контейнеризация
├── .semgrep_custom.yml # Кастомные правила статического анализа
├── .github/workflows/
│   └── ci.yml          # CI-пайплайн
├── templates/
│   └── index.html      # Страница приложения
├── static/
│   ├── css/style.css
│   └── js/main.js      # Логика анимации
└── tests/
    ├── test_algorithms.py  # Unit-тесты алгоритмов
    └── test_app.py         # Functional-тесты API
```

### Как устроены алгоритмы
Каждая функция сортировки принимает список и возвращает список «снимков»
(steps). Каждый снимок — это словарь:
```python
{
  "arr":   [...],   # полное состояние массива на этом шаге
  "cmp":   [i, j],  # индексы, которые сравниваются
  "swap":  [i, j],  # индексы, которые поменялись местами
  "done":  [...],   # индексы, уже стоящие на финальных позициях
  "cmps":  int,     # накопленный счётчик сравнений
  "swaps": int,     # накопленный счётчик свапов
}
```
Алгоритмы зарегистрированы в словаре `ALGORITHMS`, точка входа — `get_steps(arr, algorithm)`.

### API
| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/` | HTML-страница приложения |
| GET | `/api/array?size=40&mode=random` | Сгенерировать массив |
| POST | `/api/sort` | Получить шаги сортировки (JSON: array, algorithm) |

Оба API-роута обрабатывают некорректный ввод: нечисловой `size` и неизвестный
алгоритм возвращают HTTP 400 с понятным сообщением об ошибке, а не падают
с 500.

### Стек
Python 3.11 + Flask 3 (бэкенд, REST API), Vanilla JS (фронтенд, анимация),
HTML/CSS (интерфейс).

## 3. Тестирование

Применены два уровня тестирования, оба автоматизированы и подключены в CI.

**Unit-тесты** (`tests/test_algorithms.py`) проверяют алгоритмы изолированно,
без HTTP. Параметризованы по всем 7 алгоритмам — один набор кейсов проверяет
каждый. Проверяемые свойства: результат равен `sorted(input)`, мультисет
элементов не меняется, вход не мутируется, счётчики растут монотонно,
структура шага корректна. Граничные случаи: пустой массив, один элемент,
дубликаты, уже отсортированный, обратный.

**Functional-тесты** (`tests/test_app.py`) проверяют HTTP API через Flask
`test_client` — без реального сервера. Покрыты оба роута, кламп размера,
все режимы генерации, обработка ошибочного ввода (400 на нечисловой size
и неизвестный алгоритм), прогон сортировки по всем алгоритмам.

**Итого:** 31 тестовая функция, разворачивающаяся в **109 прогонов** за счёт
параметризации. Все проходят. Запуск: `pytest tests/ -v`.

В ходе написания тестов были найдены и исправлены 2 реальных бага в `app.py`:
необработанные исключения на некорректном вводе приводили к HTTP 500 —
заменены на корректный 400.

## 4. Безопасность

### Статический анализ кода (SAST)
Подключён **Semgrep** с публичными наборами `p/python`, `p/security-audit`
и тремя кастомными правилами (`.semgrep_custom.yml`). Запускается в CI с
флагом `--error` (находки валят пайплайн).

Найдено и исправлено: `app.run(debug=True)` (CWE-489, Active Debug Code —
потенциальная RCE через дебаггер Werkzeug). Исправление: debug-режим вынесен
в переменную окружения `FLASK_DEBUG`, по умолчанию выключен. Информационная
находка про `host='0.0.0.0'` осознанно подавлена с обоснованием
(нужно для работы в Docker-контейнере).

### Анализ зависимостей (SCA)
Проведён **Trivy** — сгенерирован SBOM (CycloneDX) и просканирован
Docker-образ. Найдено 156 уязвимостей: 7 в Python-пакетах, 149 в системных
Debian-пакетах базового образа.

Python-уязвимости (устаревшие pip/wheel со старыми CVE, вплоть до RCE через
зловредные wheel-пакеты) **устранены**: в Dockerfile добавлено обновление
`pip setuptools wheel` перед установкой зависимостей. Результат: 7 → 0.

Системные Debian-CVE задокументированы как остаточный риск: у большинства нет
доступных фиксов (Debian ещё не выпустил патчи), приложение к этим библиотекам
напрямую не обращается, эксплуатация большинства требует локального доступа
к контейнеру.

## 5. CI/CD

Пайплайн GitHub Actions (`.github/workflows/ci.yml`) запускается на каждый
Pull Request и push в `main`. Шаги по порядку:
1. Проверка синтаксиса (`py_compile`).
2. Тесты (`pytest`).
3. Линтер (`ruff`).
4. Статический анализ безопасности (`semgrep --error`).
5. Сборка Docker-образа.
6. (опц.) SBOM и скан образа через Trivy.

Логика: быстрые проверки раньше, дорогие позже — пайплайн падает на первой
же проблеме, не тратя время на последующие шаги.

## 6. Запуск

### Локально
```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
python3 app.py
```
Открыть `http://localhost:5000`.

### Через Docker
```bash
docker build -t sorting-visualizer .
docker run --rm -p 5000:5000 sorting-visualizer
```

## 7. Выводы

Проект прошёл полный цикл разработки, который имитирует реальную практику:
идея и описание → реализация → автоматические проверки в CI → контейнеризация
→ тестирование двух уровней → статический анализ кода → анализ зависимостей.

Что показал каждый этап:
- **Тесты** поймали 2 реальных бага в обработке ошибок, которые при ручной
  проверке легко пропустить.
- **SAST** нашёл небезопасный debug-режим Flask до того, как он мог бы уехать
  в прод.
- **SCA** показал, что собственный код и прямые зависимости чисты, а основные
  риски — в инструментах сборки (устранены) и системных библиотеках образа
  (задокументированы).

Главный практический урок: безопасность и качество — не разовая проверка
в конце, а набор автоматических ворот в CI, которые срабатывают на каждое
изменение.

### Что можно развивать дальше
- Одновременный запуск двух алгоритмов для сравнения.
- График зависимости числа операций от размера массива.
- Переход на более тонкий базовый образ (alpine/distroless) для сокращения
  поверхности атаки.
- Подключение Dependabot/Renovate для автоматического обновления зависимостей.
